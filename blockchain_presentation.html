BLOCKCHAIN FUNDAMENTALS — 01

WHAT IS A BLOCKCHAIN?

CORE DEFINITION

A blockchain is a distributed, append-only ledger — a sequence of cryptographically linked blocks, each containing data, a timestamp, and the hash of the previous block. Once written, data is computationally infeasible to alter without invalidating every subsequent block.

KEY PROPERTIES

  • Immutability — altering a block breaks the chain
  • Decentralisation — no single point of control
  • Transparency — all nodes share the same ledger
  • Consensus — nodes agree via Proof-of-Work
  • Byzantine Fault Tolerance — survives malicious nodes
GENESIS

→
BLOCK 1
00000091…3f

BLOCK N
000000b2…7a

WHY IT MATTERS

The bottleneck of traditional blockchains is sequential CPU execution — one hash at a time. Networks like Ethereum's EVM are limited in Transactions Per Second (TPS) by this constraint. Our project proves that GPU parallelism breaks this ceiling.

BLOCKCHAIN FUNDAMENTALS — 03

SHA-256 & CRYPTOGRAPHIC HASHING

WHAT IS A HASH FUNCTION?

SHA-256 maps any input to a fixed 256-bit (32-byte) digest. Deterministic, one-way, and avalanche-effect driven — a single bit change flips ~50% of output bits.

INPUT → SHA-256 → OUTPUT
"Genesis Block!" →
000000440346916553ce530e69235e945b5278abe48d98cd162ec8c62a02eb2c

WHY IT ENSURES IMMUTABILITY

Change one byte in Block 3 → its hash changes → Block 4's prevHash is now wrong → Block 4's hash changes → cascading invalidation of every subsequent block.

SHA-256 INTERNALS (64 ROUNDS)

// Our device-side implementation __device__ void sha256_device( const uchar* input, int len, uchar* output) // Macros using only bit arithmetic: ROTR(x,n) → rotate right CH(x,y,z) → choice function MAJ(x,y,z) → majority function EP0/EP1 → sigma expansions SIG0/SIG1 → message schedule

KEY INSIGHT FOR GPU

SHA-256 is pure integer arithmetic — shifts, XORs, additions. No library calls. Every thread can run a complete, independent SHA-256 using only its own registers. Zero shared state between threads.

BLOCKCHAIN FUNDAMENTALS — 04

MINING & PROOF OF WORK

THE COMPUTATIONAL PUZZLE

Mining = finding a nonce N such that:

sha256(index‖prevHash‖merkleRoot‖N)
must start with 000000…

Probability: 1 in 16⁶ ≈ 1 in 16 million per attempt. Only brute force works — no shortcut exists.

CPU BASELINE (SEQUENTIAL)

// Original findHash() — one nonce at a time for (nonce = 0; nonce < 100000000000; nonce++) { hash = sha256(header + to_string(nonce)); if (hash.substr(0,6) == "000000") return {hash, nonce}; } // Average: ~90 SECONDS

GPU APPROACH (BATCH PARALLEL)

// mineKernel — 1 thread = 1 nonce __global__ void mineKernel( const uchar* header, uint batchStart, uint* resultNonce, int* found, volatile int* abortFlag) { if (*found || *abortFlag) return; uint nonce = batchStart + blockIdx.x * blockDim.x + threadIdx.x; // compute sha256_device → check → atomicCAS }

BLOCKCHAIN FUNDAMENTALS — 05

MERKLE ROOTS & PARALLEL REDUCTION

WHAT IS A MERKLE TREE?

A binary hash tree that condenses all transaction data into a single Merkle Root stored in the block header. Tamper any transaction → root changes → block is invalid.

ROOT HASH
/ \
H(T0‖T1)
H(T2‖T3)
/ \     / \
sha256(T0)
sha256(T1)
sha256(T2)
sha256(T3)

CPU: O(N) SEQUENTIAL

Hash pairs one-by-one in a while loop. Each level waits for the previous. For 4000 transactions: slow and blocking.

GPU: O(log N) PARALLEL REDUCTION

Each round, N/2 threads hash pairs simultaneously. The array halves each round until 1 hash remains — the root.

// In-place parallel reduction __global__ void merkelKernel( uchar* hashes, int numHashes) { int i = blockIdx.x*blockDim.x + threadIdx.x; // Thread i: hash[i*2] ‖ hash[i*2+1] → hash[i] uchar* left = hashes + (i*2)*32; uchar* right = hashes + (i*2+1)*32; uchar* out = hashes + i*32; }

CUDA DYNAMIC PARALLELISM (CDP)

A parent kernel (merkleParentKernel) manages the reduction loop entirely on the device, spawning child merkelKernel grids for each level — eliminating CPU-to-GPU round-trip latency per level. Requires -rdc=true.

MINING ENGINE DEEP DIVE

KEY OPTIMISATIONS

  • __constant__ memory — block header broadcast to all warps at zero latency via L1 cache
  • Grid-Stride Loop — covers unbounded nonce space across batches
  • On-device integer-to-ASCII — eliminates PCIe transfer for nonce string conversion
  • __launch_bounds__(256) — informs compiler for optimal register allocation per SM
  • atomicCAS — lock-free, race-free result write from first winning thread

BATCH LOOP PATTERN

uint batchStart = 0; while (!h_found) { mineKernel<<<4096, 256>>>( d_header, batchStart, d_nonce, d_found, abortFlag); cudaDeviceSynchronize(); batchStart += 4096 * 256; // ~1M/batch }

ZERO-COPY INTERRUPT SYSTEM

When another node wins the block, the HTTP server thread sets networkInterrupt = true. A volatile int* mapped via cudaHostAllocMapped carries this signal directly across the PCIe bus — the GPU reads from physical RAM, bypassing L1/L2 cache. The kernel checks every 64 iterations and commits "kernel suicide" in sub-milliseconds.

CUDA STREAMS — ASYNC MINING

The mining kernel launches on a dedicated cudaStream_t, completely decoupling the GPU from the CPU. The CPU host thread remains free to receive HTTP POST /newchain requests from the network while mining runs in the background — enabling real-time network consensus without blocking.

MERKLE & VERIFICATION DEEP DIVE

MERKLE — CDP ARCHITECTURE

merkleParentKernel runs entirely on device. It loops over tree levels, spawning child merkelKernel grids via CUDA Dynamic Parallelism. Requires -rdc=true (Relocatable Device Code) in NVCCFLAGS.

// Parent kernel — device-side loop __global__ void merkleParentKernel( uchar* hashes, int count) { while (count > 1) { int pairs = count / 2; merkelKernel<<<grid, block>>>( hashes, count); // child spawn cudaDeviceSynchronize(); count = pairs; } }

VERIFICATION — KEY TECHNIQUES

  • __restrict__ pointers — guarantees no aliasing, enables coalesced memory reads and instruction reordering
  • uint32_t string comparison — 4 bytes compared per op instead of 1 (16 ops for 64-char hash vs 65)
  • atomicExch(result, 0) — fail-fast: any invalid block immediately signals global failure
  • atomicAdd(result, 0) — early-exit read without contention
  • bytesToHex device function — lookup table converts 32 hash bytes to 64 hex chars on-device

VERIFICATION STREAMS

Chain split into two halves processed on two concurrent cudaStreams. cudaMemcpyAsync of second half overlaps with kernel execution of first half — hiding PCIe transfer latency behind compute.

WHY THE BRIDGE EXISTS

nvcc cannot compile STL std::function, json.hh, or Boost headers — they use C++ template metaprogramming features that are incompatible with device compilation. The rule: .cu files may only include CUDA headers and plain C structs. All blockchain logic stays in .cpp files compiled by g++.

CUDA ACCELERATION — 01

CUDA KEYWORDS & PRIMITIVES

MEMORY HIERARCHY

__constant__ cudaHostAllocMapped cudaMemcpyAsync cudaMalloc cudaMemset cudaFree

__constant__ stores SHA-256 K[64] array and block header — broadcast from L1 cache to all threads in a warp at zero bandwidth cost. cudaHostAllocMapped creates Zero-Copy pinned RAM shared between CPU and GPU for the abort flag.

KERNEL QUALIFIERS

__global__ __device__ __launch_bounds__(256) __restrict__ volatile

__launch_bounds__(256) tells the compiler exact thread count for optimal register allocation. __restrict__ guarantees no pointer aliasing, enabling coalesced loads. volatile forces cache-bypass for abort flag reads.

ATOMIC OPERATIONS

atomicCAS atomicExch atomicAdd

atomicCAS(found, 0, 1) — only the first thread to find a valid nonce writes its result; all subsequent winners see found=1 and exit cleanly. atomicExch(result, 0) — fail-fast chain invalidation from any thread.

ASYNC & STREAMS

cudaStream_t cudaStreamCreate cudaStreamSynchronize cudaEventRecord cudaEventElapsedTime

Mining kernel on dedicated stream — CPU unblocked for HTTP. Chain verification split across 2 streams — cudaMemcpyAsync of second half overlaps with kernel on first. cudaEvent gives sub-millisecond GPU timing for all benchmarks.

CUDA ACCELERATION — 03

SHA-256 ON DEVICE

WHY REIMPLEMENT SHA-256?

OpenSSL's SHA256_Init/Update/Final are CPU library calls — the GPU cannot invoke them. We implemented SHA-256 from scratch as a __device__ function using only arithmetic operations that map directly to GPU ALU instructions.

CONSTANT MEMORY — K[64]

__constant__ uint32_t K[64] = { 0x428a2f98, 0x71374491, ... };

Stored in constant cache — broadcast to all 3,328 threads simultaneously. Eliminates 64 global memory reads per thread per hash. Critical for throughput at scale.

REGISTER-LEVEL EXECUTION

__device__ void sha256_device( const uchar* input, int len, uchar* output) { // All state in registers — no shared mem uint32_t h0=0x6a09e667, h1=0xbb67ae85... uchar msg[256]; // local buffer // Pad → 64-byte chunks → 64 rounds // ROTR/CH/MAJ/EP0/EP1/SIG0/SIG1 macros // Big-endian 32-byte output }

ZERO INTER-THREAD DEPENDENCY

Each thread's SHA-256 runs entirely in its own register file. No shared memory, no synchronisation barriers, no data hazards. This is why 3,328 threads can each run independent SHA-256 computations simultaneously with maximum SM occupancy.
